# 02 实现方法

## 一、JWT 是怎么工作的

JWT（JSON Web Token）本质上是一种**后端签发的身份凭证**。

用户登录成功后，后端会生成一段字符串返回给前端。  
前端之后每次请求接口时，都在请求头里带上：

```http
Authorization: Bearer <token>
```

后端收到后会做三件事：

1. 校验 token 是否被伪造
2. 校验 token 是否过期
3. 从 token 中解析出当前登录用户是谁、角色是什么

这样后端就不需要再靠前端传 `userId`、`role` 来判断身份。

## 二、这次项目中的认证链路

本项目现在的认证链路如下：

### 1. 登录阶段

用户请求：

- `/admin/login`
- `/teacher/login`
- `/student/login`
- `/login`

后端执行：

1. 根据用户名/姓名查用户
2. 用 `PasswordCodecService` 校验密码
3. 生成 `AuthenticatedUser`
4. 用 `JwtTokenService` 生成 JWT
5. 返回用户信息 + token

### 2. 接口访问阶段

前端访问接口时带上：

```http
Authorization: Bearer xxx.yyy.zzz
```

后端执行：

1. `JwtAuthenticationFilter` 拦截请求
2. 解析 Bearer Token
3. 校验签名和过期时间
4. 构造 `AuthenticatedUser`
5. 放入 Spring Security 上下文
6. 控制器和权限表达式从上下文中读取当前用户

## 三、为什么要用 Spring Security

如果只写一个 `JwtUtil`，那仍然只是“能解 token”。

真正有工程感的是把它接进 Spring Security 的标准链路里，因为这样可以统一解决：

- 登录态识别
- 未登录拦截
- 无权限拦截
- 角色授权
- 注解式访问控制

这也是为什么本次改造不是只加了一个工具类，而是完整引入了：

- `SecurityFilterChain`
- JWT 过滤器
- 401 / 403 处理器
- 方法级权限控制

## 四、RBAC 在这个项目里怎么做

RBAC 就是 **Role-Based Access Control**，基于角色的访问控制。

### 角色

项目定义了三种角色：

- `admin`
- `teacher`
- `student`

在代码里对应 `AppRole` 枚举。

### 控制方式

本项目主要使用两种方式：

#### 1. 角色级控制

例如：

```java
@PreAuthorize("hasRole('ADMIN')")
```

表示只有管理员能访问。

#### 2. 数据范围控制

例如：

```java
@PreAuthorize("@authz.isTeacherOrAdmin(#teacherId)")
```

表示：

- 管理员可以查任意教师数据
- 教师只能查自己的数据

学生查看班级课表、查看个人成绩，也用了同类思路。

## 五、密码兼容迁移是怎么实现的

这是这次改造里比较有技术含量、也比较贴近真实项目的一部分。

### 原始问题

老系统里：

- 有些密码是明文
- 有些用户密码可能为空
- 如果直接强切 BCrypt，会导致老账号全部无法登录

### 解决方案

当前实现采用的是**兼容迁移策略**：

1. 登录时先判断数据库中的密码是否已加密
2. 如果是已加密密码，就走 BCrypt 校验
3. 如果是明文密码，就用明文方式兼容校验
4. 兼容登录成功后，立刻把该密码改存为 BCrypt

这样做的好处是：

- 用户不用统一重置密码
- 老数据能继续使用
- 系统会逐步自然迁移到安全存储状态

## 六、为什么论坛接口也要跟着改

因为 JWT 最大的价值不是“登录能用”，而是“后端不再信任前端身份”。

论坛场景最典型：

如果前端 body 里直接传 `authorId` 和 `authorRole`，理论上任何人都可以冒充别人发帖。

所以这次改造后：

- 发帖身份从 token 中拿
- 评论身份从 token 中拿
- 点赞身份从 token 中拿
- 删除帖子时校验是否管理员或作者本人

这部分特别适合答辩时讲“为什么后端必须掌握身份主导权”。
