# 03 实现成果

## 一、现在项目已经实现的安全能力

### 1. 真实 JWT 登录

管理员、教师、学生登录后，后端返回的已经是真实 JWT，不再是演示型拼接 token。

### 2. 无状态鉴权

后端不依赖服务端 Session，接口只根据 Bearer Token 识别登录身份。

### 3. 多角色访问控制

系统已经具备：

- 管理员权限
- 教师权限
- 学生权限

并且支持按角色限制接口访问。

### 4. 数据范围隔离

不是只有“你有没有登录”，而是进一步实现了：

- 教师只能查自己的课表 / 考试
- 学生只能查自己班级的课表 / 考试
- 学生只能查自己的成绩

这类控制比单纯的登录更能体现后端能力。

### 5. 密码加密存储

新增和更新的密码已经通过 BCrypt 存储，不再继续扩散明文密码。

### 6. 历史密码平滑迁移

老账号不需要人工全量重置密码，也能逐步迁移到加密状态。

### 7. 论坛身份防伪造

论坛发帖、评论、点赞不再直接信任前端传来的作者信息。

## 二、这次改造后，项目更适合怎么写进简历

你可以把这部分写成：

> 基于 Spring Security + JWT 完成教务排课系统三角色鉴权改造，设计无状态认证链路、RBAC 接口权限控制与数据范围校验，兼容历史明文密码并实现 BCrypt 平滑迁移，提升系统安全性与工程化程度。

如果要再展开一点，可以写成：

- 负责智能排课系统后端安全体系升级，基于 Spring Security 与 JWT 构建管理员 / 教师 / 学生三角色鉴权方案
- 设计接口级 RBAC 权限控制与数据归属校验，避免教师、学生越权访问他人业务数据
- 设计密码兼容迁移机制，将历史明文密码平滑升级为 BCrypt 加密存储
- 修复论坛模块对前端身份字段的信任问题，实现后端基于登录态的身份确认

## 三、答辩时可以重点讲的点

### 1. 为什么原来的 token 不够

因为原来只是字符串拼接，后端没有真正校验签名，也没有真正的认证链路。

### 2. 为什么不用 Session

因为前后端分离项目里，JWT 的无状态认证更适合跨端调用和接口化管理。

### 3. 为什么要做 RBAC 而不是只做登录

因为真实系统里最重要的不是“能登录”，而是“不同角色只能访问自己该访问的数据”。

### 4. 为什么密码迁移比直接加密更难

因为要兼容历史存量数据，不能为了升级安全机制把旧账号全部打废。

### 5. 为什么论坛模块也要改

因为这正好体现了“后端掌控身份”的核心安全思想。
