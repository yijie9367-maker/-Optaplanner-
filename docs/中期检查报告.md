课题主要任务：
本课题旨在设计并实现一套基于人工智能技术的大学智能课程排课辅助系统，以提升高校教学管理效率并优化课程资源配置。围绕该总体目标，主要任务包括：

需求分析与系统总体设计
对高校课程排课业务流程进行调研与分析，明确系统功能需求与性能要求，完成系统总体架构设计与模块划分。

排课模型与算法设计
研究课程排课问题的约束条件与优化目标，构建符合高校教学实际的排课数学模型；基于OptaPlanner约束求解引擎设计排课优化算法，通过Constraint Streams API定义硬约束与软约束规则。

系统功能模块实现
在系统平台上实现课程信息管理、教师与学生信息管理、教室与班级管理、排课约束配置、自动排课、课表可视化与导出等核心功能模块，并扩展考试管理、成绩管理、消息公告与论坛交流等教务辅助功能。

冲突检测与优化机制设计
实现排课过程中的冲突检测功能，包括时间冲突、教师冲突、教室冲突、班级冲突及教室容量超限冲突，并通过约束权重动态配置与求解参数调整优化排课结果，提高排课合理性与满意度。

多角色权限与安全认证设计
基于JWT + Spring Security + RBAC模型实现管理员、教师、学生三种角色的登录认证与权限隔离，确保系统数据安全与操作合规。

系统测试与效果评估
通过功能测试、性能测试与应用测试验证系统的稳定性与实用性，并对排课效果进行评估，总结系统在实际应用中的优势与不足。

论文撰写与成果总结
对系统设计过程、关键技术与实验结果进行系统整理与总结，形成完整的毕业论文。

1.简述开题以来所做的具体工作和取得的进展或成果
自开题以来，在指导教师的指导下，围绕"基于AI的大学智能课程排课辅助系统"的研究目标，结合既定技术路线，系统性开展了以下工作，并取得了阶段性成果：

文献调研与技术路线确定
查阅并分析了国内外有关高校课程排课系统、教学辅助系统、遗传算法在排课优化中的应用、前后端分离系统架构及排课约束求解引擎等方向的相关文献共15篇。通过文献对比与技术评估，最终确定以OptaPlanner约束求解引擎为核心排课优化工具，采用Spring Boot + Vue 3的前后端分离模式作为系统实现框架，以MySQL 8.0为数据库，JWT + Spring Security为认证方案。

系统总体设计与功能模块划分
在充分分析高校排课业务需求的基础上，完成了系统的总体架构设计。后端采用Controller → Service → Repository → Entity/DTO/VO五层分层架构，前端采用视图（views）→ 组件（components）→ 组合式函数（composables）→ API模块（api）→ 工具函数（utils）的分层结构。明确了系统由基础信息管理、智能排课、约束配置、课表可视化、考试管理、成绩管理、消息公告、论坛交流及多角色权限认证等多个子系统组成。完成了系统用例图、功能结构图、数据库E-R图及业务流程设计，为后续开发奠定基础。

后端架构搭建与安全认证设计
基于Spring Boot 2.7框架完成了后端系统的基础环境搭建，集成Spring Security实现角色权限控制，采用JWT（JJWT 0.11.5）实现无状态认证机制。设计了SecurityFilterChain → TokenFilter → UserDetailsService的认证链路，通过@PreAuthorize注解实现接口级角色控制，Service层注入当前用户身份实现数据级权限隔离。实现了管理员、教师、学生三种角色的独立登录入口，用户密码采用BCrypt加密存储。

数据库设计与基础数据管理
依据业务需求设计了MySQL 8.0关系型数据库（utf8mb4编码），完成教师、学生、班级、教室、课程、排课结果、考试、成绩、消息、论坛帖子、论坛评论、论坛投票、约束配置、管理员等14张核心数据表的构建。使用Spring Data JPA完成实体映射与数据访问层代码编写，支持分页查询与模糊搜索。

管理员后台页面搭建与基础信息CRUD实现
已完成管理员端全部基础信息管理页面的搭建与功能实现，具体包括：
① 教师管理：增删改查、分页、模糊搜索、Excel批量导入导出；
② 学生管理：增删改查、分页、模糊搜索、Excel批量导入导出；
③ 班级管理：增删改查、分页、Excel批量导入导出；
④ 教室管理：增删改查、分页、Excel批量导入导出；
⑤ 课程管理：增删改查、分页、Excel批量导入导出；
⑥ 管理员账号管理：增删改查、分页、Excel批量导入导出。
前端基于Vue 3 + Element Plus + Axios实现，提炼了useTableOperation、useFormDialog、useDataLoading等可复用组合式函数，以及apiFactory、formatters等通用工具模块，大幅提升了代码复用率。所有基础信息页面已与后端接口完成联调，CRUD操作稳定可用。

排课求解模块设计与实现
已完成OptaPlanner 8.44.0排课求解模块的核心设计与编码。具体包括：
① 领域模型构建：设计Lesson（课程）、Timeslot（时间段）、Room（教室）、TimeTable（课表）等规划实体与规划变量类；
② 约束规则编写：采用Constraint Streams API定义了硬约束（教师时间冲突、教室时间冲突、班级时间冲突、教室容量超限）和软约束（课程分散度、周次偏好、时间槽偏好、课程连续性、避免晚间排课）；
③ 约束权重动态配置：实现了ConstraintConfig实体、Repository、Service与Controller，支持通过前端界面对各软约束的权重和求解参数（solverSeconds等）进行CRUD管理，权重数据在求解时动态注入OptaPlanner求解器；
④ 求解接口实现：通过SolverManager实现异步求解任务管理，后端提供求解触发、状态查询与结果获取的REST API。

使用模拟教学数据（20+教师、30+班级、15+教室、50+课程）进行了排课求解测试，约束求解引擎可在设定时间内生成无硬冲突的可行课表方案，初步验证了OptaPlanner技术路线的可行性。

前端排课页面与可视化开发
完成了管理员排课管理前端页面的开发，包括约束配置管理界面（各软约束权重滑块与求解参数设置）、一键智能排课操作入口。开发了ScheduleVisualization课表可视化组件，支持按教室、按教师、按班级三个维度切换展示课表，能够标注不同课程的时间占用情况。

阶段总结与后续计划安排
总体来看，目前已基本完成系统设计、后端架构搭建、安全认证框架、数据库建设、全部基础信息管理的CRUD功能及排课求解核心模块的开发。管理员后台页面已全面可用，排课算法已初具模型并通过模拟数据验证。整体进度符合开题计划要求。

2.存在的具体问题
在课题研究与系统开发过程中，虽然已取得一定进展，但由于时间、经验和技术条件等因素的限制，当前仍存在以下几个方面的问题与不足：

排课求解效果仍需进一步调优
目前OptaPlanner求解器已能生成无硬冲突的排课方案，但在软约束综合优化方面仍有提升空间。约束权重配置虽已支持动态调整，但各权重参数的最佳取值范围尚未经过充分的实验验证，面对不同规模的教学数据时，求解参数（solverSeconds、算法策略选择）的自适应调整能力不足。

课表可视化功能尚不完善
当前ScheduleVisualization组件已实现基础的三维度课表切换展示，但冲突高亮标注功能尚未完全实现，排课结果中的冲突课程（经手动调整后可能产生的新冲突）无法直观呈现。拖拽式手动微调排课的功能尚未开发。

教师端与学生端功能尚未开发
目前系统仅完成了管理员端页面的搭建与功能实现，教师端（课表查询、考试管理、成绩录入、同事列表、授课班级）和学生端（个人课表查看、成绩查询、论坛浏览）的前端页面及对应后端接口尚未建设。

考试管理与论坛等功能处于开发初期
考试管理、成绩管理、消息公告的后端Service/Controller代码已编写，但前端管理界面尚未完全对接。论坛子系统后端实体、Repository与Service已基本完成，前端论坛页面仍待开发。

系统集成与联调不充分
各模块目前仍处于相对独立的状态，前后端接口联调尚未覆盖全部业务流程（如排课结果确认后持久化到数据库、排课变更的消息通知等），系统整体联动性和业务闭环仍需加强。

权限控制的数据级隔离待完善
当前基于@PreAuthorize的角色级权限已实现，但Service层的数据范围隔离（如教师仅能查看本人课表、仅能修改本人帖子）仅在部分模块落实，部分模块的数据权限过滤逻辑仍需补全。

系统测试工作尚不充分
目前仅对基础CRUD和排课求解进行了基本的正确性验证，尚未系统性开展功能测试、接口测试、多角色场景测试和性能测试。Excel导入导出在万级数据量下的性能表现尚未评估。

文档与代码规范化工作仍需完善
部分技术文档、接口文档尚未整理成型，代码注释覆盖率有待提升。
