# AI智能排课系统——开发总结与心得

## 一、项目概述

本项目是一个基于AI的大学智能排课系统配套学生端微信小程序，旨在为大学生提供便捷的课表查看、考试查询、论坛交流和AI智能助手服务。整个开发过程从需求分析到测试运维，深度借助了多种AI工具，是一次完整的"人+AI协作"软件工程实践。

**项目规模**：5大功能模块 | 6个页面 | 12个API接口 | 47条测试用例 | 4份配套文档

**技术栈**：Spring Boot 2.7 + 微信原生框架 + MySQL + JWT + OptaPlanner + 阿里云百炼

**开发周期**：约2周（课余时间）

## 二、开发流程回顾

### 2.1 需求分析

最初只有一个模糊的想法——"做一个学生能查课表的小程序"。将想法用自然语言描述给DeepSeek后，它帮我将零散的思路整理成了结构化的需求文档：5大功能模块的功能表、完整的数据结构定义、10个API接口清单、页面结构和交互细节。

这个过程让我意识到：**AI不是来替代思考的，而是把模糊的想法翻译成结构化的方案**。它不会追问你"为什么要这个功能"、"这个边界条件怎么处理"——这些问题必须人来想清楚。AI的角色更像一个需求分析师助理，帮你整理你已经想好的东西，但不会帮你发现问题。

### 2.2 原型设计

借助Stitch工具，将需求文档快速转化为高保真的小程序UI原型。相比传统流程中先画线框图、再手动写UI代码的方式，Stitch直接将设计稿转换成了可运行的WXML+WXSS代码。配合DeepSeek对细节的调整（统一配色#4A90D9、优化布局、适配不同屏幕尺寸），6个页面的原型在半天内就达到了可演示的水平。

**核心体会**：原型设计阶段是最能体现"AI加速"的环节。传统方式下从设计到可交互原型通常需要3-5天，而AI辅助下这个时间被压缩到了半天。因为AI消除了"设计→代码"这个最大的摩擦力。

### 2.3 编码开发

编码阶段是AI介入最深、也是踩坑最多的环节。

**前端开发**方面，微信小程序的WXML+WXSS+JS是我不太熟练的技术栈。DeepSeek在这里发挥了极大作用：将需求文档和API文档一并提供给它，它能生成完整的页面骨架代码。遇到问题时，将报错信息和相关代码粘贴进去，通常能在几轮对话内定位并修复。最典型的例子是课表网格算法——7列×5行×20周的数据匹配与渲染，DeepSeek一次性生成了可用的版本。

**后端开发**方面，Spring Boot是我相对熟悉的技术栈，AI的作用更多体现在效率提升上。Controller-Service-Repository三层代码、SecurityConfig配置、JWT认证过滤器等，都由DeepSeek根据需求描述批量生成，人工只做审查和微调。

但AI也制造了不少麻烦：它将baseUrl写死为localhost，导致真机调试时网络请求全部失败；它推荐的百炼API调用路径是错误的，返回404；它建议的后端直调方案因为认证问题（ACCOUNT_NOT_LOGIN）根本行不通。每一次踩坑都在提醒我：**AI产出的代码需要人的审校，AI提出的方案需要人的判断**。

**AI助手集成**是整个项目中最特殊的一个环节——在一个AI辅助开发的小程序里，再嵌入另一个AI作为产品功能。最终采用的方案是通过web-view加载阿里云百炼AppFlow SDK，SDK自行处理认证和对话逻辑，避免了后端代理的复杂性和安全性问题。这个过程也让我深刻体会到了"AI给的方案不一定最优，人要判断哪个更可行"。

### 2.4 测试

将需求和代码交给DeepSeek，设定角色为"资深测试工程师"，让它生成了47条功能测试用例，覆盖登录、课表、考试、论坛、个人中心、AI助手全部6个模块，以及12个API的curl测试命令和异常场景（网络异常、数据异常、并发场景）。

关键技巧是**角色扮演**：说"帮我测试"只会得到模糊的建议；说"作为资深测试工程师，为以下API编写包含正常、边界、异常的测试用例"，得到的才是结构化的专业输出。AI的输出质量，很大程度上取决于你给它设定的"角色"和"任务边界"。

### 2.5 文档生成

文档生成是AI表现最出色的环节。将项目代码、需求文档和测试用例一起提供给DeepSeek，它自动生成了4份结构化文档：《需求文档》、《AI智能体操作指引手册》、《测试与运维手册》、《PPT生成提示词》。

**为什么文档生成是AI最强的环节？** 因为文档本质上是对已有信息的结构化重组，不需要创造新逻辑。这正是大语言模型最擅长的——理解已有内容，按照指定的格式重新组织输出。传统开发中，文档编写往往是开发完成后最痛苦的环节，但在AI辅助下，它变成了"点一下按钮"的事情。

## 三、AI工具链使用心得

### 3.1 工具链配置

| 阶段 | 工具 | 评价 |
|------|------|------|
| 原型设计 | Stitch | 设计→代码的桥梁，最省时间的环节 |
| 需求/编码/测试/文档 | DeepSeek V4 Pro | 全流程主力，贯穿开发始终 |
| AI助手（项目内置） | 阿里云百炼 | 产品功能而非开发工具，加分项 |
| PPT | 豆包 | 提示词→演示文稿，汇报最后一公里 |

### 3.2 DeepSeek V4 Pro深度使用感受

DeepSeek V4 Pro是本次开发中使用频次最高的工具，几乎贯穿了全部环节。对它的使用体会有几点：

**最强的能力**：结构化内容生成。无论是需求文档、代码框架、测试用例还是运维手册，只要给足上下文，它的输出质量都相当高。尤其是Java/Spring Boot这种成熟技术栈，生成代码的可用率能达到80%以上。

**最明显的短板**：缺乏项目全局视野。它能写好单个文件，但不知道这个文件在整个项目中的位置；它能生成正确的代码，但不知道这段代码在你的运行环境中能否正常工作。所以会出现localhost写死、API路径错误、方案不适用等问题。

**最有效的使用方式**：不是把它当"外包程序员"用（"帮我做个课表系统"），而是当"结对编程伙伴"用（"这是我当前的文件，我要实现XX功能，帮我写一下"）。上下文越具体，输出越准确。一次给200行上下文比一次给20行效果好，但一次让它写200行代码不如分10次写20行。

### 3.3 跨语言学习的意外收获

除了小程序本身，这次还体验了用DeepSeek学习Python——一门我从未接触过的语言。传统学习路径是"先学语法再做项目"，AI辅助下的路径变成了"先出项目再学语法"。

一周内完成视频爬虫、自动连点器、文件批处理等多个实用脚本。**关键不是AI帮我写了多少代码，而是每生成一段代码，我都会追问"这段为什么这样写"、"那个库是干嘛的"**。AI既是代码生成器，也是随时可用的老师。

这种"以用促学"的模式，效率远超传统的"先学后用"。第一天就在写真实项目，遇到不懂的当场问，学了立刻用，学习闭环从几个月压缩到了几天。

## 四、核心认知：人+AI协作的本质

### 4.1 AI是什么角色

经过这次实战，我对AI在软件开发中的定位有了清晰的认知：**AI是一个不知疲倦、学富五车、但缺乏判断力的全栈实习生**。

它勤奋：可以连续生成几千行代码不喊累。它博学：Spring Boot、JWT、微信小程序、OptaPlanner，几乎什么技术栈都能写。但它没有判断力：它不知道你真正需要什么、不知道什么方案在你的环境下可行、不知道哪些代码是"对的"哪些只是"能跑的"。

所以人+AI协作的本质是：**人做决策，AI做执行**。技术选型你来定，架构设计你来定，业务边界你来定；AI帮你写代码、写测试、写文档。方向盘在人手里，AI是发动机。

### 4.2 5条实战法则

1. **先喂上下文，再下指令**——把需求文档、现有代码、错误日志都给AI，比直接说"帮我写"准确10倍
2. **小步快跑，逐段验证**——让AI一次写50行，验证通过再继续，比一次写500行高效得多
3. **报错即粘贴，不要自己猜**——把完整报错+相关代码一起给AI，30秒定位 vs 自己查30分钟
4. **角色扮演，精准引导**——"你是资深测试工程师""你是Spring Boot架构师"，输出质量天差地别
5. **人决策，AI执行**——选什么技术栈、用什么架构，必须人定。AI不会为你的项目负责

### 4.3 AI的"能"与"不能"

**AI擅长**：文档生成、CRUD代码、已知bug修复、测试用例、模板化内容、结构化信息重组。

**AI不擅长**：零上下文推理、架构决策、环境配置、理解业务边界、运行时调试、跨文件依赖管理。

**核心认知**：AI产出代码的速度远超人类，但产出的代码质量参差不齐。未来程序员的竞争力不在于"写代码的速度"，而在于**判断AI产出质量的能力**——能一眼看出AI写的代码哪里有问题、哪里可以优化、哪里需要重写。

## 五、个人收获

### 5.1 思维方式的转变

这次经历最大的收获，不是做出了一个小程序，而是建立了一种全新的开发思维：**遇到问题先想"AI能不能帮我"，而不是"我要花多少时间学"**。

这种思维下，学习变成了一个"即时满足"的过程：想爬视频？DeepSeek写代码，我读代码学原理。想写连点器？同上。想做小程序？搭工具链，开工。

### 5.2 对"全栈"的重新理解

以前对"全栈"的理解是"一个人会前端+后端+数据库+部署"。现在对"全栈"的理解是"一个人+AI工具链 = 一个团队的能力"。

不是你会多少种语言，是你知道在什么阶段用什么工具、怎么给AI下指令、怎么判断AI的产出质量。**AI时代的全栈，工具链就是你的团队**。

### 5.3 持续探索的方向

这次实践验证了AI辅助开发的可行性，但也暴露了当前工具的局限性。未来值得探索的方向包括：AI如何更好地理解项目全局上下文、如何实现跨文件的协同修改、如何在代码生成时自动考虑运行环境差异。在这些问题被解决之前，"人在回路上"（Human in the loop）仍然是不可替代的。
