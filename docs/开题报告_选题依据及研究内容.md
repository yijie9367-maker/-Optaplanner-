1) 选题依据

本选题"基于AI的大学智能课程排课辅助系统"的立项依据充分，可从技术可行性、现实需求和政策支持三个层面加以论证。

技术层面，Spring Boot 2.7框架以其自动配置和起步依赖特性大幅简化了Java Web应用的构建过程；Vue 3 + Element Plus的前端技术栈提供了组件化、响应式的界面开发能力；OptaPlanner作为成熟的约束求解引擎，将排课问题建模为规划实体与规划变量的组合优化问题，通过Constraint Streams API定义硬约束（教师/教室/班级时间冲突、教室容量限制）与软约束（课程分散度、时间槽偏好等），并支持约束权重动态配置。上述技术均具有成熟社区支持和丰富工程实践积累，为系统实现提供了可靠的技术保障。

现实需求层面，随着高校招生规模持续扩大和课程体系日趋复杂，传统人工排课模式面临效率低下、冲突频发、资源利用率不足等突出痛点。一个班级的课表调整可能引发连锁反应，教务人员往往需要数天甚至数周反复修改才能达到可接受方案。同时，考试安排与排课结果紧密关联，成绩管理与教学质量评价也需要统一的数据支撑，教务管理的各个环节呼唤一体化、智能化的解决方案。市场对具备约束可配置、结果可视化、数据可导出的智能排课工具存在迫切需求。

政策层面，《教育信息化2.0行动计划》明确要求推动信息技术与教育管理深度融合，建设智慧校园。本项目以人工智能技术赋能教务管理核心环节，契合国家教育信息化发展战略，具备良好的政策支持环境。

2) 主要研究内容

本研究内容涵盖系统功能设计、技术实现方案及平台性能与安全三大板块。

一、系统主要功能

1. 多角色用户管理：实现管理员、教师、学生三种角色的独立登录与权限分离。管理员拥有系统最高权限，可进行排课、考试管理、消息发布与论坛管理；教师可查看课表、管理考试与成绩、参与论坛交流；学生可查看个人课表与考试成绩、参与论坛讨论。采用JWT + Spring Security + RBAC模型实现角色级权限控制和数据级权限隔离。

2. 基础信息管理：实现教师、学生、班级、教室、课程、管理员等基础教学资源的增删改查操作，支持分页查询、模糊搜索和批量操作。提供Excel模板下载、批量数据导入与数据导出功能，使用EasyExcel实现高性能读写。

3. 智能排课核心功能：以OptaPlanner约束求解引擎为核心，将排课问题建模为含7项硬约束和多种软约束的组合优化问题。硬约束涵盖教师时间冲突、教室时间冲突、班级时间冲突和教室容量限制；软约束包括课程分散度、周次偏好、时间槽偏好、课程连续性、避免晚间排课等维度。管理员可通过前端界面对各软约束的权重进行动态调整，并设置求解时间（solverSeconds）等参数。系统提供一键智能排课、冲突检测及求解结果统计功能。

4. 约束配置管理：提供约束权重的CRUD管理界面，约束配置数据在排课求解时实时注入OptaPlanner求解器，使教务人员可根据不同学期的实际情况灵活调整优化目标偏好，无需修改代码即可改变排课策略。

5. 课表可视化与导出：前端按教室、按教师、按班级三个维度提供可视化课表视图，冲突课程以高亮方式标注。排课结果可导出为Excel，支持按维度筛选打印。

6. 考试与成绩管理：实现考试安排的增删改查（含字段完整性、时间逻辑、关联数据存在性、唯一性等多层校验），支持成绩批量录入与统计查询，采用"先查全班学生再左连接已有成绩"策略提供完整成绩视图。支持Excel导入导出。

7. 消息公告与论坛交流：管理员可向教师和学生分类发布消息公告。论坛子系统支持发帖、评论、点赞点踩功能，投票操作覆盖同类型取消、异类型切换、无记录新增三种路径，管理员具备帖子置顶与内容管理权限。

二、技术实现方案

1. Spring Boot后端架构：采用Spring Boot 2.7 + Spring Security + Spring Data JPA搭建后端服务，遵循Controller → Service → Repository → Entity/DTO/VO的五层分层架构。Controller层负责REST接口暴露与@PreAuthorize权限校验；Service层封装业务逻辑与校验链；Repository层基于JPA实现数据访问与自定义查询；Entity层映射数据库表结构，DTO/VO层负责数据传输与视图呈现。统一使用ResponseResult封装API响应。

2. 智能排课算法设计：采用OptaPlanner 8.44.0的Constraint Streams API定义约束流规则。硬约束（教师/教室/班级时间冲突、教室容量）在求解过程中必须满足，软约束通过可配置权重进行多目标权衡。求解器配置采用模拟退火+禁忌搜索的组合策略，通过SolverManager实现求解任务的异步管理。

3. 数据库设计：使用MySQL 8.0存储结构化数据（utf8mb4编码），设计教师、学生、班级、教室、课程、排课结果、考试、成绩、消息、论坛帖子、论坛评论、论坛投票、约束配置、管理员等14张核心数据表，通过外键关联与冗余字段设计平衡查询性能与数据一致性。

4. 前后端分离开发：后端通过RESTful API提供数据接口，前端基于Vue 3 + Vite + Element Plus构建SPA应用，使用Axios实现前后端数据交互。axios拦截器自动注入JWT令牌，Vue Router路由守卫实现角色级页面访问控制。前端采用视图（views）→ 组件（components）→ 组合式函数（composables）→ API模块（api）→ 工具函数（utils）的分层结构。

三、平台性能与安全

1. 系统性能优化：前端采用Vite构建工具实现快速冷启动与HMR热更新；后端通过JPA延迟加载与分页查询控制数据库负载；Excel导入导出采用EasyExcel的流式读写避免OOM。排课求解过程异步执行，前端轮询展示求解进度。

2. 全方位安全防护：使用Spring Security + JWT实现无状态认证，JWT令牌包含用户ID与角色信息，设置合理过期时间。后端通过@PreAuthorize注解实现接口级角色控制，Service层注入当前用户身份实现数据级权限隔离（如教师只能查看自己的课表、只能修改自己的帖子）。用户密码采用BCrypt加密存储。支持Docker Compose三容器编排（MySQL + 后端 + 前端Nginx）实现容器化部署，通过Nginx反向代理统一对外暴露端口。

3) 研究思路及方案

本研究将按照软件工程开发流程，分阶段推进系统的设计与实现工作。

1. 需求分析阶段：通过文献调研深入了解排课问题的研究现状、主流求解方法及国内外教务系统的功能特点。结合高校教务管理的实际场景，梳理管理员、教师、学生三类用户的功能需求。明确排课的核心硬约束（不可违反）和软约束（可优化）范围，设计用例图与需求规格说明。

2. 系统设计阶段：基于需求分析结果，完成系统总体架构设计。采用前后端分离模式，后端划分为Controller、Service、Repository、Entity/DTO/VO五层，前端划分为视图、组件、组合式函数、API模块、工具函数、样式、常量和路由配置八层。设计数据库E-R图与14张核心数据表的字段结构、索引策略。将排课问题建模为OptaPlanner求解模型，设计规划实体（Lesson）、规划变量（Timeslot、Room）与约束流规则。设计JWT认证流程（SecurityFilterChain → TokenFilter → UserDetailsService）与RBAC权限模型。制定统一的API响应格式和前后端接口规范。

3. 技术选型阶段：确定技术栈——后端Java 17 + Spring Boot 2.7 + Spring Security + Spring Data JPA + OptaPlanner 8.44.0 + EasyExcel 3.3.4 + JJWT 0.11.5；前端Vue 3 + Vite + Element Plus + Pinia + Vue Router 4 + Axios；数据库MySQL 8.0；构建工具Maven + npm/Vite；部署方案Docker Compose三容器编排。

4. 功能开发阶段：按照系统设计文档，分模块迭代开发。先搭建项目框架，完成认证模块与基础CRUD；随后重点开发OptaPlanner排课求解模块（领域模型构建 → 约束规则编写 → 约束配置管理 → 求解接口对接）；再依次实现考试管理、成绩管理、消息公告和论坛子系统；最后完成前端课表可视化、冲突高亮和数据导出功能。每个模块开发过程中编写单元测试，确保代码质量。

5. 系统测试阶段：在功能开发完成后，进行全面的功能测试与集成测试。验证三种角色的登录认证、权限隔离和数据范围控制是否正确。使用真实教学数据构造排课实例，测试不同约束权重配置下的求解效果（硬约束违规数、软约束评分、求解时间）。验证Excel导入导出在万级数据量下的性能表现。根据测试结果进行缺陷修复与性能优化。

6. 部署与文档阶段：编写Dockerfile与docker-compose.yml，实现MySQL + 后端Spring Boot + 前端Nginx三容器的一键编排部署。撰写项目开发文档、API接口文档和部署说明。整理测试报告，完成毕业论文的撰写与答辩准备。
